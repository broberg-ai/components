# F039.5 — Session-key reset tooling (the promised "ask components to reset it" flow)

> Story under **F039 — Auto-enrollment service (Discovery write-layer)**. From fd-sundhed #18311 (2026-07-21).

## Motivation

Discovery's enroll auth is **trust-on-first-use**: the first `POST /api/enroll` for a session binds `sha256(key)` in the `session_keys` table; later enrolls must present the same key. `bindSessionKey` uses `ON CONFLICT(session) DO NOTHING`, so a binding is never overwritten. That is the anti-hijack property — a leaked session key can only write THAT session's row, never the whole roster.

But a session that **loses its key** (missing/lost `.env`, as fd-sundhed did) is permanently locked out of re-enrolling: a freshly-generated key mismatches the bound hash → `session_key_mismatch` 401. The 401 message already PROMISES the escape hatch — *“… or ask components to reset it”* (`server.ts:392`) — but **no reset tooling exists**. It was handled once (2026-07-21) with a one-off manual DB delete run inside the Fly machine (cleared fd-sundhed's `session_keys` row; its 7 enrollments in the separate table were preserved). That manual surgery is the drift this story removes.

## Scope

- `resetSessionKey(session)` on `EnrollStore` → `DELETE FROM session_keys WHERE session = ?`. Touches ONLY the key binding; the `enrollments` table (the actual adoptions) is a separate table and is left intact.
- A committed `apps/discovery/scripts/reset-session-key.mjs <session>` that **verify-before-deletes**: prints the binding it will clear + the session's preserved enrollments, deletes only that session's key row, prints the after-state.
- `Dockerfile` COPYs the script into the image so it runs via `flyctl ssh`.

### Non-goals / security

- **NO network-reachable reset endpoint.** An admin-gated `POST /api/enroll/reset` would be a network-reachable **TOFU-bypass attack surface** — the whole point of TOFU is that a key can't be reset remotely. The reset stays **owner-only over `flyctl ssh`** (physical control of the Fly machine = the auth).
- Single-session targeted only — never a bulk/all-sessions reset.
- No change to the enroll/bind logic itself.

## How to run

```
flyctl ssh console -a broberg-discovery -C 'cd /app/apps/discovery && bun scripts/reset-session-key.mjs <session>'
```
The session then re-binds its new key on the next `POST /api/enroll`.

## Reuse

This makes an already-PROMISED capability real + auditable + reusable, instead of ad-hoc SQL surgery each time — the harness-contract 'wire the mechanism' applied to the enroll write-layer. No external dependency; it uses the existing `@libsql/client` + env creds already in the machine.

## Rollout

Store method + test → script → Dockerfile COPY → `bun run typecheck && bun run test` in apps/discovery → deploy → (the flow is already documented by the enroll 401 message that points here).
