# F008.8 — mountAuth/getSession compose with plugin-narrowed createTypedAuth (no cast)

> Story under **F008 — @broberg/auth**. Follow-on defect from **F008.7** (which introduced `createTypedAuth`).

## Motivation

`createTypedAuth` (F008.7) returns a **plugin-narrowed** `Auth<{ …plugins:[magicLink, passkey] }>` so plugin-augmented `api.*` methods are statically available with no cast. But `mountAuth(app, auth: Auth, …)` in `packages/auth/src/hono.ts` takes the **wide** `Auth` (= `Auth<BetterAuthOptions>`). Better Auth's `Auth<O>` is **invariant** in `O` (its `adapter`/`$context` appear in both co- and contravariant positions), so `Auth<{plugins:[…]}>` is **not assignable** to `Auth<BetterAuthOptions>`.

Result: the *exact* combination the README recommends — `createTypedAuth` + `mountAuth` — fails `tsc`. Reported by **happy-little-place** (#18069); runtime is perfect (`/api/auth/get-session` 200, full magic-link round-trip), so this is **types only**. Their interim workaround is an `as unknown as` cast (their `apps/api/src/app.ts`, commit c2887c5) — the drift this package exists to remove.

## Scope

- Widen `mountAuth` + `getSession` (`packages/auth/src/hono.ts`) to accept the **structural minimum** they actually use — `auth.handler` for `mountAuth`, `auth.api.getSession` for `getSession` — instead of the invariant wide `Auth`. Both `createAuth` (wide) and `createTypedAuth` (narrowed) instances then mount with **no cast**.
- `getSession` becomes generic over the auth instance so the **precise** (plugin-augmented) session return type is preserved.
- Add a **compile-only type-level test fixture** that REDs `pnpm typecheck` if the param reverts to the invariant `Auth`.
- Check the `/next` adapter (`toNextHandler`) for the same invariance issue; fix symmetrically if present.
- Bump `0.1.1 → 0.1.2`, publish via the `auth-v0.1.2` OIDC tag (existing name → token-free, no OTP), remove any cast note from the README.

### Non-goals

- **No runtime change** — only the exported param types widen; the one-liner bodies are untouched.
- Don't touch `createAuth` / `createTypedAuth` themselves.
- No new dependency.

## Architecture sketch

`mountAuth` only ever calls `auth.handler(c.req.raw)`. `getSession` only ever calls `auth.api.getSession({ headers })`. So the helpers need a structural slice, not the nominal `Auth`:

```ts
// mountAuth: accept anything with a Web-standard handler
function mountAuth(
  app: Hono,
  auth: { handler: (request: Request) => Response | Promise<Response> },
  basePath = "/api/auth",
): void

// getSession: generic over the instance → precise session type preserved
function getSession<A extends { api: { getSession: (o: { headers: Headers }) => unknown } }>(
  c: Context,
  auth: A,
): ReturnType<A["api"]["getSession"]>
```

Both `createAuth(...)` and `createTypedAuth(...)` structurally satisfy these, so both mount without a cast, and the invariance wall is gone because we no longer name the invariant `Auth<O>` in the parameter position.

## Dependencies

None. Better Auth types unchanged; no other `@broberg/*` package touched.

## Reuse

This IS reuse-first in action: happy-little-place filed the gap (#18069) instead of hand-rolling a local mount, and we **extend `@broberg/auth`** so every consumer's `createTypedAuth` + `mountAuth` composes — the whole fleet benefits from one patch. No Discovery search applies (we are fixing the owned package itself, not sourcing a new capability). Precedent: F008.7 (same package, same 'extend don't cast' spirit) and `@broberg/webpush.sendSilent()` (consumer-filed gap → package grew).
