# F044.1 — @broberg/speech-dictionary edit API (components' half of the cardmem editor)

**Status:** proposed — pending cardmem's confirmation they'll build the UI half (F185-adjacent settings surface).
**Owner:** `components` (this endpoint; the git-commit + version-bump + tag-push + npm-publish mechanics belong to whoever owns the package, same as every other `@broberg/*`).
**Depends on:** F044 (`@broberg/speech-dictionary` v0.1.0, already built) and its Trusted Publisher being set up (same bootstrap-then-OIDC path as every other package this cycle).
**Requested by:** Christian, in two steps — first "let mig redigere + auto-publish ved gem" (no live-API alternative — ruled out, breaks Trail's offline on-device use case), then "jeg orker ikke flere auth sider og user logins," then "kan det redigeres i cardmem et sted, det er jo en del af vores fleet og SDLC."

## Motivation
Christian wants to add/fix dictionary entries without touching a terminal or waiting on a cc session, and without a new login screen. He's already authenticated in cardmem daily (it's the fleet's SDLC hub) — editing from there means zero *new* auth surface for him, even though the actual git/npm mechanics still need to happen somewhere. That "somewhere" has to be components (the package owner), not cardmem, so the two repos' release pipelines don't get tangled.

## Scope (in)
1. **A single authenticated write endpoint on Discovery** (components' one live service): `POST discovery.broberg.ai/api/speech-dictionary/edit`.
   - Auth: a dedicated shared secret (`SPEECH_DICT_EDIT_KEY`), header `x-speech-dict-key` — service-to-service only, never a user-facing login. Narrower scope than the existing `DISCOVERY_ENROLL_KEY` (that one's for self-reported package adoption; this one can mutate a git repo, so it gets its own key).
   - Body is a **diff, not a full-file replace** — `{ addTerms?: TermEntry[], removeTerms?: string[], addCorrections?: CorrectionEntry[], removeCorrections?: string[] }` — so concurrent edits don't clobber each other and commits stay readable (one clear diff per save, not a giant file rewrite).
2. **Mechanics on save:** read current `packages/speech-dictionary/data/{terms,corrections}.json` from `broberg-ai/components` via the GitHub Contents API, apply the diff, bump `packages/speech-dictionary/package.json`'s patch version, commit both files + the version bump, create + push a `speech-dictionary-v<version>` tag. No local git clone needed on the Discovery Fly machine — GitHub's Contents/Git-refs API does file+commit+tag over HTTPS.
3. **The existing tag-triggered OIDC publish workflow does the rest** (once Trusted Publisher is set up for this package) — no OTP, no login, same as every other `@broberg/*` release.
4. **Response** — `{ ok: true, version, tag, commitSha }` or `{ ok: false, error }`, so cardmem's UI can show a clear "published as v0.1.2" confirmation (or a real error) rather than a bare 200.

## Scope (out)
- **No UI.** That's cardmem's half (F185-adjacent or a new small fleet-settings surface — cardmem's call where it lives in their own information architecture).
- **No generic "edit any file in any repo" API.** This endpoint is narrowly scoped to `speech-dictionary`'s two data files + its own version — not a general components-repo-write backdoor. A future second editable package would get its own narrowly-scoped endpoint, not a parameterized any-file version (YAGNI until a second real case exists).
- **No optimistic-locking / conflict resolution beyond "last write wins."** Single editor (Christian) today; not worth building for concurrent-edit races that don't exist yet.

## Contract cardmem needs to build against
```
POST https://discovery.broberg.ai/api/speech-dictionary/edit
Headers: x-speech-dict-key: <secret components issues to cardmem>
Body: {
  "addTerms"?: [{ "term": string, "group": "product"|"person"|"brand"|"tech" }],
  "removeTerms"?: [string],
  "addCorrections"?: [{ "wrong": string, "right": string, "note"?: string }],
  "removeCorrections"?: [string]  // by "wrong" value
}
Response 200: { "ok": true, "version": "0.1.2", "tag": "speech-dictionary-v0.1.2", "commitSha": "..." }
Response 4xx/5xx: { "ok": false, "error": string }
```
