# F033.7 — Shared reusable `fly-server-deploy` GitHub Actions workflow

## Motivation
`@broberg/deploy-core` (F033) covers the **static-site / incremental content-sync** deploy pattern (`flyLiveDeploy`). It does **not** deploy a Bun/Hono **server app** with a custom Docker image. happy-little-place hit this (#18003) and fell back to the canonical `flyctl deploy --remote-only` in a hand-rolled `ci.yml`. Many fleet repos hand-copy that same workflow — that copy-paste **is** the drift the shared inventory exists to kill.

## Decision — a reusable workflow, NOT a JS function
Server-app deploy is a **CI concern**, and `flyctl deploy` is already the shared primitive (a CLI). So the reuse win is a **reusable `workflow_call` GitHub Actions workflow** the whole fleet calls — not a `flyServerDeploy()` npm export. It lives in `components` at `.github/workflows/fly-server-deploy.yml`.

## Scope (the DEPLOY half)
The caller owns the **test gate** (`deploy: needs: test`). This workflow does:
1. **Ship-dark guard** — no `FLY_API_TOKEN` -> skip deploy, job stays green.
2. **Optional host build** (`build_command` + `build_runtime` node|bun) — the Fly remote-builder can't run esbuild (EPIPE), so build the dist on the runner and COPY it in the Dockerfile.
3. **`flyctl deploy --remote-only`** (config / app / dockerfile inputs).
4. **Health-verify** — poll `health_url` for HTTP 200 (fail if never 200): prove it serves, not just that deploy returned.
5. **Optional required-secrets presence check** — assert named Fly secrets exist (`flyctl secrets list`, names only, values never printed).

### Non-goals
- Not the test/build gate itself (caller's `needs`).
- Not the static content-sync path (that stays `flyLiveDeploy`).
- No secret values ever printed or interpolated into a shell command (read via `env`).

## Consumer usage
```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    steps: [ ... your tests ... ]
  deploy:
    needs: test
    uses: broberg-ai/components/.github/workflows/fly-server-deploy.yml@main
    with:
      app: my-app
      config: fly.toml
      health_url: https://my-app.fly.dev/health
      build_command: pnpm install --frozen-lockfile && pnpm build   # optional (esbuild-on-host)
      build_runtime: node                                            # none|node|bun
      required_secrets: DATABASE_URL,SESSION_SECRET                   # optional
    secrets:
      FLY_API_TOKEN: ${{ secrets.FLY_API_TOKEN }}
```

## Rollout
- **Consumer #1 = happy-little-place** — lifts their proven `ci.yml` onto `uses:` and reports the first real run (the runtime proof; this plan ships YAML-validated only).
- Add a **Fly infra tip** on Discovery so the next repo finds it instead of hand-rolling.
- Not runtime-verified until consumer #1's first deploy through it — stated explicitly, no checkmark before then.

## Reuse
- **Reused:** the fleet's own `flyctl` + the OIDC/publish-workflow conventions already in `.github/workflows/`; happy-little-place's `ci.yml` as the seed. No new dependency.
- **Built (this):** the shared workflow file — the one home for server-app Fly deploy so no repo hand-copies it again. Discovery-tip pointer added for discovery.
