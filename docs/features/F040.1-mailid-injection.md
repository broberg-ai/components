# F040.1 — MailID injection: stamp a per-project correlation code on every outbound mail

> **Parent:** F040 (`@broberg/mail-templates`) · **Owner:** components · **Counterpart:** cardmem F131 (ingest) · **Status:** Backlog (gated on Christian's go for the cardmem side)
> **Origin:** Christian's idea, worked out with cardmem over intercom #5731–#5740 (2026-06-22).

## The product picture

Every cardmem project gets a unique, unguessable **MailID**. The shared mail service stamps it onto **every outbound mail** for that project. Then Christian's `cb@webhouse.dk` inbox (which cardmem already reads) **sorts itself by project** — and the same code lets cardmem filter its **own E2E test mails**. Zero manual setup: he never types a code.

This generalises a pattern already hand-rolled in the small (sanne's `SA-LENS-E2E` subject marker for reading its test mails back) into one fleet primitive.

## Split of work

| Side | Owner | What |
|---|---|---|
| **Injection** (this card) | components | stamp the `Ref: <MailID>` line on every outbound mail |
| **MailID generation + storage** | cardmem (F131) | auto-generate an opaque MailID per project, store in project config (one source) |
| **Routing / matching** | cardmem (F131) | F131.17 full-body fetch + a **body-scope** keyword-watcher whose keyword = the MailID |

The watcher row (`kind=keyword`, `scope=body`, `keyword=<MailID>`) **is** the code→project mapping — no separate correlation table needed.

## Components-side design (this card)

Injection lives in **`@broberg/mail`'s send-path**, not only the F040 templates — because the sender is the single point **every** outbound mail passes through (templated *and* ad-hoc magic-link / notification mail). That is what makes "on ALL outbound" actually true (Christian: "et API til at indsætte koden i ALLE outbound mails"). F040 templates inherit the placement.

- `@broberg/mail` gains an opaque **`mailId`** option (mailer config and/or per-send). Absent → unchanged behaviour (backward compatible, ship-dark).
- On send with `mailId` set, append a **discreet `Ref: <mailId>` line** to **both** the HTML body and the plain-text part.
- **The carrier is body text, on purpose.** It must survive a reply. Two things that do NOT work, both confirmed with cardmem:
  - A custom header (`X-Cardmem-MailID`) is **not echoed** by the replying client — only the quoted *body* survives a reply. Header would only help E2E / direct reads, not replies.
  - `display:none` / white-on-white / zero-width text gets **stripped on reply** AND **hurts spam score**. So the line is *discreet* (small, faint grey) but *real* text.
- Gmail's "…" quote-collapse does not remove the quoted text from the raw MIME, so a real text line in the quoted original is still present for cardmem's full-body matcher.

## Non-goals

- **Not** address/`+tag`/`Reply-To` routing. Considered; needs DNS (cardmem.com MX) + a dedicated mailbox + DWD scope, and cardmem reads `cb@webhouse.dk` not cardmem.com. **Deferred** as a documented future upgrade that would additionally capture replies where the user deletes the entire quote (the one case the body line misses).
- **Not** the MailID generation, the watcher auto-provisioning, or the ingest matching — those are cardmem's F131 side.
- **No** new template language; the line is appended, not a `{{token}}`.

## Anti-spoof

The MailID is an **opaque random token** (e.g. `CM-<random>`), not the project slug — unguessable, so a third party cannot write the code into a mail to inject into a project. cardmem additionally gates ingest on the F167 trusted-sender allowlist (`mockup_share_recipients`). Defence in depth.

## Dependencies / sequencing

- cardmem **F131.17** (full-body fetch: `format=full` → MIME-walk → base64url-decode → cap) is the real prerequisite for the routing to fire — today ingest only reads `format=metadata` + the ~200-char snippet (the top of the mail), and the Ref line sits at the bottom. This card's injection is independent and can ship first, but end-to-end routing waits on F131.17.
- The mail watcher surface is "inert until the Gmail service is configured" (cardmem infra) — does not touch this card.

## Rollout

Ships as a backward-compatible minor of `@broberg/mail` (existing, shipped v0.1.0). Inert until a `mailId` is passed, so no behaviour change for current consumers. F040 templates pick up the placement when that engine lands.
